How to read this

Diagram 4 · Runtime flow

A simplified app execution pathLuồng thực thi app đã giản lược

Purpose: show how a shared UI definition becomes platform output at runtime.

Mục đích: cho thấy một định nghĩa UI dùng chung trở thành output theo nền tảng khi chạy.

Spec: nodes = compiled app, generated InitializeComponent, dependency properties/events, layout/rendering, runtime target; edges = runtime sequence; layout = left-to-right sequence; confidence = medium-high, simplified from architecture and runtime project names.
Why this matters: most control logic lives above the runtime split, so a fix in shared layout/control code can affect every target.
Boundary: platform-specific files and runtime projects translate shared behavior into DOM, Skia rendering, or native views.
Vì sao quan trọng: phần lớn logic control nằm phía trên điểm tách runtime, nên một fix trong layout/control dùng chung có thể ảnh hưởng mọi target.
Boundary: file theo platform và runtime project dịch hành vi dùng chung sang DOM, Skia rendering hoặc native views.

Diagram 5 · Build and tooling flow

How contributors usually validate changesCách contributor thường validate thay đổi

Purpose: connect the repository's build configuration, generators, sample app, runtime tests, and DevServer/RemoteControl tooling.

Mục đích: nối cấu hình build, generator, sample app, runtime tests và tooling DevServer/RemoteControl của repository.

Spec: nodes = crosstargeting props, solution filters, MSBuild/source generators, outputs, tests, samples, DevServer; edges = build/validate sequence; layout = left-to-right workflow; confidence = high from AGENTS.md and project files.
  • For fast local validation, the repo guidance favors platform-specific solution filters and Skia desktop when possible.
  • Source generators and MSBuild tasks are part of the build product, so generated output issues often need generator-level fixes.
  • Runtime/UI changes should be validated with the closest RuntimeTests path, not only compilation.
  • Để validate local nhanh, hướng dẫn repo ưu tiên solution filter theo platform và Skia desktop khi có thể.
  • Source generator và MSBuild task là một phần của sản phẩm build, nên lỗi output sinh ra thường cần fix ở tầng generator.
  • Thay đổi runtime/UI nên được validate bằng RuntimeTests gần nhất, không chỉ compile.

Evidence and confidence

What this visualization is based onVisualization này dựa trên điều gì

Observed evidence

  • Project families such as src/Uno.UI, src/Uno.UWP, src/Uno.Foundation, src/Uno.UI.Runtime.Skia*, and src/Uno.UI.Runtime.WebAssembly.
  • Solution filters including Uno.UI-Skia-only.slnf, Uno.UI-Wasm-only.slnf, Uno.UI-UnitTests-only.slnf, and mobile/Windows filters.
  • Source generator files such as XamlFileGenerator.cs, XamlCodeGenerator.cs, and DependencyObjectGenerator.cs.
  • Validation surfaces in SamplesApp, Uno.UI.RuntimeTests, and Uno.UI.Tests.

Bằng chứng đã quan sát

  • Các nhóm project như src/Uno.UI, src/Uno.UWP, src/Uno.Foundation, src/Uno.UI.Runtime.Skia*, và src/Uno.UI.Runtime.WebAssembly.
  • Các solution filter gồm Uno.UI-Skia-only.slnf, Uno.UI-Wasm-only.slnf, Uno.UI-UnitTests-only.slnf, và filter mobile/Windows.
  • Các file source generator như XamlFileGenerator.cs, XamlCodeGenerator.cs, và DependencyObjectGenerator.cs.
  • Các bề mặt validation trong SamplesApp, Uno.UI.RuntimeTests, và Uno.UI.Tests.

Confidence notes

High confidence for package names, high-level layers, solution filters, and generator/tooling presence. Runtime flow is simplified and should be read as an onboarding model, not a precise call graph.

Ghi chú về độ tin cậy

Độ tin cậy cao cho tên package, lớp kiến trúc mức cao, solution filter và sự tồn tại của generator/tooling. Luồng runtime đã được giản lược và nên đọc như model onboarding, không phải call graph chính xác.
